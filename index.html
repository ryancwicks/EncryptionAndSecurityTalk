Password Managers

Using easy to remember passwords or reusing passwords is a really bad practice. One compromised account can mean all of your accounts are compromised.

Password managers store your account and passwords securely in an encrypted file only you can open.

Pro's

  • Only remember 1 password
  • Can use randomly generated, long, unique and unguessable passwords for each account.
  • Easy syncing across devices without giving all your data to Google or Apple with some services.
  • No more saving passwords and forms in a browser.

Con's

  • Single point of failure (it had better have a good password)
  • You are trusting the company who provides the service.
  • Sometimes inconvenient.
  • Can be expensive.

Two Factor Authentication

Logins require you to provide two different bits of information to log into account. These are usually:

  • Something you know:
    • A password
  • Something you have:
    • A cellphone (SMS, one time password)
    • A two factor device (Yubikey, Google Titan)
    • A body (biometrics)
    • A public/private key pair

VPN's

Virtual Private Network encrypts all of your network traffic from you device to a server somewhere in the world (not just http traffic). It's been used historically for accessing internal company networks from outside the company, but is often used to prevent eavesdropping on all traffic leaving a computer.

It might not be needed for everyone, but definitely handy when using public wifi, or when travelling.

Key Exchange Methods

Symmetric encryption is great for encrypting data being sent between two parties, but how do we let the other party know what the key is?

  • Out of band methods:
    • Courier
    • Phone call
    • Diplomatic bag
  • In band methods:
    • Diffie-Hellman key exchange
    • Public Key Architecture (Certificate Authorities)
    • Web of Trust

Diffie-Hellman Key Exchange

Pro's

  • Keys can be generated over insecure connections (secure against eavesdroppers).
  • Perfect forward secrecy (a new key is generated every session, or multiple times a session).

Con's

  • Susceptible to man-in-the-middle attacks.

Asymmetric Encryption

Encryption that requires two keys. Either can be used to encrypt a block of data, but you can't decrypt with the same key. Only the other key can be used to decrypt it.

This private public key system is very powerful and allows you to do interesting things

Anyone can access Alice's public key to encrypt a block of data, but only Bob can decrypt it.

Don't lose control of that private key.

Is this really Alice's public key?

Per session generated public key pairs can be used as part of the Diffie-Hellmen key exchange.

This can be a problem if both private keys are eventually discovered. You lose perfect forward secrecy.

Am I really talking to Bob?

Alice can use her private key to sign a message that Bob can then verify could only have been signed by Alice.

You can mathematically prove to a third party that you are in possession of the private key.

But how do you know the public key you used the verify actually belongs to who you think it does?

Public Key Architecture

One method we can use to verify someones identity is to ask a Certificate Authority if you are talking to who you think you are talking to.

At the start of a connection, a request is made by the client to a Certificate Authority, trusted by all parties, for a digital certificate which serves for other users as a non-tamperable authentication of identity.

This list of certificate authorities is built into your web browser of operating system.

They are a sort of online notary that says "This public key belongs to this user."
